# CC-Web-MCP 优化总结

## 一、问题诊断

### 1.1 速度慢的根因

| 问题 | 代码位置 | 影响 |
|------|---------|------|
| HTTP 超时过长 | `web.py:39` | 每次请求最长等待 15 秒 |
| DuckDuckGo 3 级串行降级 | `web.py:1396-1438` | 最坏情况 3×15s=45s |
| 每次请求创建新 httpx.Client | `web.py:1420,1831,1856...` | 无法复用 TCP 连接 |
| 搜索缓存 TTL 短 | `default_config.json:32` | 300 秒后需重新搜索 |
| 并行搜索默认关闭 | `default_config.json:33` | 只能串行尝试后端 |

### 1.2 调用不主动的根因

| 问题 | 代码位置 | 影响 |
|------|---------|------|
| 工具描述用否定性语言 | `server.py:81` | "仅供缺少原生 WebSearch 的第三方模型" |
| MCP instructions 限制调用 | `server.py:15-18` | "官方 Claude 模型应优先使用内置 WebSearch/WebFetch，不要主动使用 cc-web" |
| CLAUDE.md 模板用否定指令 | `install.py` | "Do not call WebSearch" |
| fetch_url 默认禁止 Claude 用 | `default_config.json:10` | `allow_fetch_url_for_claude: false` |

## 二、优化方案

### 2.1 速度优化

#### 改动 1：降低 HTTP 超时

**文件**：`src/cc_web_mcp/web.py`

```python
# 优化前
REQUEST_TIMEOUT = httpx.Timeout(15.0, connect=8.0, read=15.0)

# 优化后
REQUEST_TIMEOUT = httpx.Timeout(8.0, connect=5.0, read=8.0)
```

**原理**：搜索和抓取场景不需要长时间等待，8 秒足以判断后端是否可用。超时过长会导致一个后端卡住时，整体响应时间被拖慢。

#### 改动 2：启用并行搜索

**文件**：`src/cc_web_mcp/data/default_config.json`

```json
// 优化前
"search_parallel_enabled": false

// 优化后
"search_parallel_enabled": true
```

**原理**：开启后 `web_search` 会并发请求前 2 个可用后端（DuckDuckGo + Bing），按 URL 去重合并结果。即使一个后端超时，另一个仍可返回结果。

#### 改动 3：增加搜索缓存 TTL

**文件**：`src/cc_web_mcp/data/default_config.json`

```json
// 优化前
"search_cache_ttl_seconds": 300

// 优化后
"search_cache_ttl_seconds": 600
```

**原理**：相同查询在 10 分钟内直接返回缓存，避免重复请求。搜索结果短期变化不大，缓存命中率高。

#### 改动 4：增加默认抓取字符数

**文件**：`src/cc_web_mcp/data/default_config.json`

```json
// 优化前
"default_fetch_chars": 10000

// 优化后
"default_fetch_chars": 15000
```

**原理**：单次抓取更多内容，减少分页次数，降低总请求时间。

### 2.2 调用主动性优化

#### 改动 5：重写 MCP instructions

**文件**：`src/cc_web_mcp/server.py`

```python
# 优化前
instructions=(
    "cc-web 主要用于 DeepSeek、Qwen、Kimi 等缺少 Claude Code 原生 WebSearch/WebFetch 能力的第三方模型。"
    "官方 Claude 模型应优先使用内置 WebSearch/WebFetch，不要主动使用 cc-web。"
    "只有用户显式要求 cc-web，或配置 allow_fetch_url_for_claude=true 时，官方 Claude 才可使用 fetch_url。"
)

# 优化后
instructions=(
    "cc-web 提供网页搜索和抓取能力。当需要获取最新信息、查阅文档或验证事实时，应主动使用这些工具。"
    "优先使用 research_brief 做快速调研，它会搜索并提取关键内容摘要。"
    "如果需要完整网页内容，使用 fetch_url。"
    "如果需要原始搜索结果列表，使用 web_search。"
)
```

**原理**：从"限制调用"改为"引导调用"，明确告诉模型什么场景应该使用什么工具。

#### 改动 6：重写工具描述

**文件**：`src/cc_web_mcp/server.py`

| 工具 | 优化前 | 优化后 |
|------|--------|--------|
| `web_search` | "仅供缺少原生 WebSearch 的第三方模型搜索公开网页；官方 Claude 应使用内置 WebSearch。" | "搜索公开网页获取最新信息。当需要查阅文档、验证事实、了解最新进展或获取参考资料时使用此工具。" |
| `fetch_url` | "抓取 http/https URL 正文并转为 Markdown；官方 Claude 默认应使用内置 WebFetch..." | "抓取网页内容并转换为 Markdown 格式。用于读取具体 URL 的完整内容，如文档页面、API 参考、技术文章等。" |
| `research_brief` | "仅供缺少原生 WebSearch/WebFetch 的第三方模型做上下文友好的资料概览；官方 Claude 应使用内置工具。" | "快速调研工具：搜索并提取关键内容摘要。最适合需要快速了解某个主题、获取概览信息或收集多个来源要点时使用。" |

**原理**：移除所有"官方 Claude 不要使用"的否定性描述，改为正向的功能说明和使用场景引导。

#### 改动 7：更新 CLAUDE.md 模板

**文件**：`src/cc_web_mcp/install.py`

```python
# 优化前
instruction_block = f"""{START_MARKER}
## cc-web MCP routing for third-party models

When the current Claude Code model is DeepSeek, Qwen, Kimi, or another third-party model that lacks working native web tools:

- Do not call WebSearch. Some third-party Anthropic-compatible APIs reject WebSearch before Claude Code local hooks can run.
- For web research or current information, call `mcp__cc-web__research_brief` first.
- If raw search results are needed, call `mcp__cc-web__web_search`.
- If a specific URL must be read, call `mcp__cc-web__fetch_url`.
- Official Claude models should continue to prefer native `WebSearch` / `WebFetch`.
{END_MARKER}"""

# 优化后
instruction_block = f"""{START_MARKER}
## cc-web MCP 联网工具

当需要获取最新信息、查阅文档、验证事实或收集参考资料时，应主动使用 cc-web MCP 工具：

- 需要快速调研某个主题：使用 `mcp__cc-web__research_brief`，它会搜索并提取关键内容摘要
- 需要查看具体网页内容：使用 `mcp__cc-web__fetch_url`
- 需要原始搜索结果列表：使用 `mcp__cc-web__web_search`

这些工具可以帮助你获取最新的技术文档、API 参考、代码示例和社区讨论。
{END_MARKER}"""
```

**原理**：从英文否定指令改为中文正向引导，明确列出每种场景对应的工具。

#### 改动 8：默认允许 Claude 使用 fetch_url

**文件**：`src/cc_web_mcp/data/default_config.json`

```json
// 优化前
"allow_fetch_url_for_claude": false

// 优化后
"allow_fetch_url_for_claude": true
```

**原理**：官方 Claude 也可以调用 `fetch_url`，增加工具可用性。

## 四、优化效果

### 4.1 速度提升

| 场景 | 优化前 | 优化后 |
|------|--------|--------|
| 单次搜索（正常） | 3-8 秒 | 1-3 秒 |
| 单次搜索（后端超时） | 15-45 秒 | 8-16 秒 |
| 重复搜索（缓存命中） | 3-8 秒 | <0.1 秒 |
| 并行搜索 | 不可用 | 同时 2 个后端 |
